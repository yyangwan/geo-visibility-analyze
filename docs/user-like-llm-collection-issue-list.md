# 真实用户仿真采集 Issue 列表

本文档把 [实施任务拆解](./user-like-llm-collection-implementation-tasks.md) 进一步拆成 issue 级别，方便直接建卡、排期和分工。

## Epic 1: 平台配置与数据模型

### Issue 1.1: 新增平台配置表

目标：
- 让每个平台的搜索倾向、参数和解析规则可配置。

工作内容：
- 新增 `platform_configs` 表。
- 定义配置 JSON 结构。
- 支持版本号和回滚字段。

依赖：
- 无。

验收：
- 能读写平台配置。
- 配置可按平台独立保存。

### Issue 1.2: 扩展平台响应记录表

目标：
- 完整保存平台原始响应和搜索元数据。

工作内容：
- 为 `platform_response_records` 增加原始留存字段。
- 覆盖 search 状态、search query、reasoning、search results、raw payload。

依赖：
- Issue 1.1 可同时或先后进行。

验收：
- 一条平台响应能完整还原原始信息。

### Issue 1.3: 评估是否拆分 raw_collection_records

目标：
- 判断是否需要把原始归档从现有响应表中进一步拆出。

工作内容：
- 评估现有表扩展后的容量。
- 如果字段继续膨胀，设计新表。

依赖：
- Issue 1.2。

验收：
- 给出明确的“先扩展现有表”或“新增原始表”结论。

## Epic 2: 平台适配器与采集

### Issue 2.1: 适配器读取平台配置

目标：
- 让查询行为按平台配置执行。

工作内容：
- 为适配器补充配置读取能力。
- 查询前加载平台配置。
- 将配置注入请求构造。

依赖：
- Issue 1.1。

验收：
- 每个平台能按自己的配置发起请求。

### Issue 2.2: 统一增强 PlatformResponse

目标：
- 让响应对象承载完整归档与分析所需字段。

工作内容：
- 保留 citations、token、model、finish reason。
- 增加原始 payload / search metadata 承载能力。

依赖：
- 无。

验收：
- 单条响应对象足够驱动后续归档。

### Issue 2.3: Kimi 多轮 search 处理

目标：
- 处理 Kimi 的 tool call / echo 流程。

工作内容：
- 单独处理 `web_search`。
- 保存中间 tool call。
- 合并最终答案与搜索元数据。

依赖：
- Issue 2.2。

验收：
- Kimi 也能保存完整原始响应链路。

### Issue 2.4: 审计任务接入原始采集

目标：
- 把平台一致性采集接进现有审计流程。

工作内容：
- 审计任务发起平台查询。
- 保存原始响应。
- 保持现有分析链路运行。

依赖：
- Issue 2.1, Issue 2.2。

验收：
- 审计结束后能查到原始响应和分析结果。

## Epic 3: 响应解析

### Issue 3.1: 引用提取器

目标：
- 提取不同平台的 citations。

工作内容：
- 支持数字引用、来源字段、markdown 链接等格式。
- 按平台配置提取规则。

依赖：
- Issue 1.1, Issue 2.2。

验收：
- 至少覆盖当前主流平台引用格式。

### Issue 3.2: 搜索状态提取

目标：
- 识别是否触发搜索和搜索词。

工作内容：
- 提取 search enabled / search triggered。
- 提取 search query / search metadata。

依赖：
- Issue 1.1, Issue 2.2。

验收：
- 能区分未搜索、搜索但无关键词、搜索且有完整关键词三种情况。

### Issue 3.3: 解析降级与错误记录

目标：
- 解析失败不阻断采集。

工作内容：
- 捕获解析异常。
- 记录 parse_error。
- 保留 raw_response。

依赖：
- Issue 3.1, Issue 3.2。

验收：
- 解析失败时审计流程仍能结束。

## Epic 4: 原始归档

### Issue 4.1: 原始响应落库

目标：
- 把平台原始响应完整写入数据库。

工作内容：
- 保存 response_text。
- 保存 raw_response 和 raw_response_text。
- 保存 citations、search metadata、request params。

依赖：
- Issue 1.2, Issue 2.2。

验收：
- 单次响应可完整还原。

### Issue 4.2: 原始记录去重与引用关系

目标：
- 防止重复写入污染分析。

工作内容：
- 保持 `audit_id + prompt_id + platform` 唯一约束。
- 分析表通过外键引用原始记录。

依赖：
- Issue 4.1。

验收：
- 同一响应只落一条原始记录。

## Epic 5: 分析层收敛

### Issue 5.1: 分析只读原始记录

目标：
- 确保分析不写回原始采集层。

工作内容：
- 分析任务只读取 `platform_response_records`。
- 结果写入 `response_analyses` 或现有分析表。
- 保持原始记录不可变。

依赖：
- Issue 4.1, Issue 4.2。

验收：
- 原始记录不会被分析任务改写。

### Issue 5.2: 保持现有分析输出不变

目标：
- 前端现有分析结果内容和模块保持不变。

工作内容：
- 保持品牌提及、来源权威、竞品、趋势、建议等输出结构。
- 避免引入前端新模块。

依赖：
- Issue 5.1。

验收：
- 前端无需做结构性调整。

## Epic 6: 平台一致性校准

### Issue 6.1: 平台参数校准

目标：
- 让采集结果更接近目标平台 UI。

工作内容：
- 为每个平台调参。
- 记录默认搜索倾向。
- 校准引用和结果偏好。

依赖：
- Issue 1.1, Issue 2.1。

验收：
- 平台 UI 对照差异可接受。

### Issue 6.2: 对照样本集

目标：
- 建立一致性验证样本。

工作内容：
- 准备 10 到 20 个 query。
- 覆盖事实、选购、对比、方案、解释问题。
- 保存平台 UI 结果样本。

依赖：
- Issue 6.1。

验收：
- 样本可重复对照。

## Epic 7: 前端边界

### Issue 7.1: 保持前端现有展示不变

目标：
- 前端展示模块、内容和交互保持现状。

工作内容：
- 不新增原始对照页。
- 不新增平台校准页。
- 沿用现有报告、建议、趋势展示。

依赖：
- 无。

验收：
- 前端用户只看到现有分析结果。

### Issue 7.2: 必要时补充说明文案

目标：
- 避免前端用户误解此次改动。

工作内容：
- 仅在必要时调整少量说明文案。
- 不改模块结构。

依赖：
- Issue 7.1。

验收：
- 用户不会把后台一致性校准误认为新前端功能。

## Epic 8: 测试

### Issue 8.1: 单元测试

目标：
- 覆盖配置、解析和归档核心逻辑。

工作内容：
- 配置读取测试。
- citation 提取测试。
- search 状态提取测试。
- raw_response 存储测试。

依赖：
- Issue 2.1, Issue 3.1, Issue 3.2, Issue 4.1。

验收：
- 核心函数有稳定测试。

### Issue 8.2: 集成测试

目标：
- 覆盖审计全链路。

工作内容：
- 审计创建到结束。
- 单平台失败不影响其他平台。
- 归档和分析双写边界验证。

依赖：
- Issue 4.1, Issue 5.1。

验收：
- 审计任务全链路可跑通。

### Issue 8.3: 对照测试

目标：
- 验证平台一致性。

工作内容：
- 与平台 UI 手工结果对照。
- 对比引用链路和搜索状态。
- 记录差异原因。

依赖：
- Issue 6.2。

验收：
- 能稳定复现平台差异。

## 3. 推荐开工顺序

1. `1.1` 平台配置表
2. `1.2` 响应记录表扩展
3. `2.1` 适配器读取配置
4. `4.1` 原始响应落库
5. `3.1` / `3.2` 解析器
6. `2.3` Kimi 多轮处理
7. `5.1` 分析只读原始记录
8. `6.1` / `6.2` 平台一致性校准
9. `8.1` / `8.2` / `8.3` 测试

## 4. 前端边界声明

- 前端现有分析结果内容和模块不变。
- 不新增原始平台对照页。
- 不把平台搜索一致性暴露给前端用户。
- 平台一致性校准和原始数据分析都属于后台工作。
