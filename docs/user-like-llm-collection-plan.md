# 真实用户仿真采集方案 V3

## 目标

核心目标不是做一个“更聪明的分析器”，而是模拟大模型自有平台的原生搜索行为，例如豆包 APP、DeepSeek APP/网页、Kimi、文心一言、通义千问等，让系统输出尽可能贴近真实用户在这些平台里实际看到的搜索结果。

同时，系统要把一次搜索的全部原始内容完整保存下来，供后续分析复用。这里的“全部内容”包括但不限于：
- 最终回答文本
- 搜索状态和搜索触发信息
- 分析过程或推理过程
- 引用网页、引用列表、来源元数据
- 搜索结果列表及其排序
- 模型名称、版本、参数、token 使用情况
- 平台返回的全部原始字段

本方案不把重点放在搜索提示词的采集上。提示词生成方式可以继续保持现状，即 AI 生成或人工录入，但不要让“提示词采集”干扰采集主目标。

## 方案定位

这套系统分成两层：

1. **采集层**：尽量像真实用户在自有平台上发起搜索，得到最接近平台原生体验的结果。
2. **分析层**：基于已保存的原始结果做后续分析，不反向改写原始记录。

换句话说，采集层关注“像平台”，分析层关注“可分析”。两者必须分离。

## 为什么要改

现有链路更像“分析工具链”，不是“真实用户视角的搜索记录”。

典型问题有三类：

1. **采集行为偏分析化**
   - 搜索输入容易被改写成更利于分析的形式
   - 搜索开关、参数、输出格式容易被人为干预
   - 结果看起来更规整，但离真实用户看到的内容更远

2. **原始证据保留不完整**
   - 只保留最终回答，不保留推理、引用、来源、搜索状态
   - 平台返回的丰富字段被压缩或丢弃
   - 后续很难复盘平台到底返回了什么

3. **平台差异没有被原样保留**
   - 不同平台的搜索习惯、引用格式、结果偏好差异明显
   - 如果强行统一，会丢失平台特征

## 设计原则

1. **平台原生优先**
   - 优先复现平台默认的搜索行为
   - 尽量不对平台行为做额外干预

2. **原始结果完整留存**
   - 任何原始字段都尽量保留
   - 解析失败也不能丢原始数据

3. **采集与分析严格分层**
   - 原始数据只追加，不重写
   - 分析结果单独存储，不回写原始记录

4. **少改输入，多保输出**
   - 查询尽量保持用户原话
   - 不为了结构化分析去改写输入

5. **平台差异显式建模**
   - 不同平台保留各自的搜索偏好、引用样式和元数据结构

6. **可对照验证**
   - 结果要能和平台 UI 手工搜索对照，验证是否“像真的”

## 目标范围

### 需要重点做的

- 模拟平台原生搜索行为
- 完整保存单次搜索的全部原始内容
- 保留搜索状态、引用、分析过程、来源网页、token 和元数据
- 支持平台差异化处理
- 支持后续分析基于原始数据展开

### 不作为重点的

- 不重点优化搜索提示词采集
- 不把采集输入强行改造成分析型 prompt
- 不追求统一所有平台的响应格式
- 不把分析结果回写为原始结果

## 推荐架构

### 总体链路

```text
用户原话 / 手动录入 / AI 生成
          ↓
平台意图识别与配置选择
          ↓
按平台原生方式发起搜索
          ↓
捕获完整原始响应
          ↓
原始归档 + 结构化解析
          ↓
独立分析与聚合
```

### 1. 平台配置层

这一层为每个平台维护一份“原生行为配置”，重点不是统一，而是保真。

配置内容建议包括：

- 默认搜索行为：默认是否搜索、什么情况下触发搜索
- 搜索强度：偏搜索、弱搜索、可选搜索、强制搜索
- 输出偏好：引用格式、是否返回推理、是否返回搜索状态
- 参数策略：temperature、top_p、max_tokens 等
- 响应识别规则：用于识别引用、搜索状态和平台特有字段

示例：

```yaml
deepseek:
  search:
    mode: optional
    default_enabled: true
  parameters:
    temperature: 0.2
    max_tokens: 4096
  capture:
    citation_patterns:
      - "\\[(\\d+)\\]"
    preserve_fields:
      - reasoning
      - citations
      - search_metadata

doubao:
  search:
    mode: strong
    default_enabled: true
  parameters:
    temperature: 0.3
    max_tokens: 2048
  capture:
    citation_patterns:
      - "来源：(.+)"
      - "\\[(\\d+)\\]"
    preserve_fields:
      - search_results
      - source_urls
      - video_sources
```

### 2. 采集执行层

这一层负责把一次查询送进目标平台，并尽可能得到和真实用户一致的结果。

采集执行建议遵循以下规则：

- 输入保持自然语言原话
- 默认单轮请求，避免历史上下文污染
- 不附加“请用 JSON 返回”等分析约束
- 不为了规范化结果而强行改写平台输出
- 平台默认搜索行为优先，不人为扭曲触发逻辑
- 平台返回什么，就尽量原样接住什么

### 3. 原始归档层

这一层保存“用户看到的第一手结果”，核心原则是完整、可追溯、不可逆改写。

建议保存的字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `trace_id` | 同一次请求的追踪 ID |
| `platform` | 平台标识 |
| `provider_model` | 实际模型名 |
| `query_intent` | 查询意图 |
| `user_query` | 用户原始输入 |
| `search_triggered` | 是否触发搜索 |
| `search_query` | 实际搜索词，如平台返回 |
| `response_text` | 完整回答文本 |
| `citations` | 引用列表 |
| `reasoning_process` | 推理/分析过程 |
| `search_results` | 原始搜索结果 |
| `token_usage` | token 使用情况 |
| `finish_reason` | 结束原因 |
| `request_params` | 请求参数 |
| `request_id` | 平台请求 ID |
| `created_at` | 时间戳 |
| `raw_response` | 原始完整 JSON |

这一层的要求很简单：

- 不重写
- 不聚合
- 不二次生成
- 不丢字段
- 解析失败也必须保留原文

### 4. 响应解析层

这一层只做“可分析化”，不参与原始留存。

职责包括：

- 识别平台响应格式
- 提取引用和来源
- 提取搜索状态
- 提取平台特有元数据
- 在解析失败时降级保留原始响应

注意：

- 解析结果是辅助数据，不是原始证据
- 原始响应始终保留为事实来源
- 平台差异不要在这里被抹平

### 5. 分析层

分析层继续保留现有分析能力，例如：

- 品牌提及分析
- 来源权威分析
- 内容结构分析
- 战略分析
- 竞品对比

但必须明确一条边界：

- 分析只能基于已保存的原始数据
- 分析结果独立存储
- 分析结果不能回写原始采集记录

## 平台策略

不同平台的搜索行为差异很大，建议按平台配置不同原生策略，而不是统一模板。

### DeepSeek

- 偏推理
- 搜索可选，不宜强制扭曲
- 常见引用格式为数字引用
- 适合保留推理和来源链路

### 豆包

- 偏强搜索
- 结果更可能包含字节系内容
- 视频、图文等来源特征更明显
- 重点保留搜索状态和来源元数据

### Kimi

- 长文本能力强
- 搜索后往往输出更长的综合结果
- 常见来源可能偏公众号和长文
- 重点保留长文本、引用和原始链接

### 文心一言

- 百度生态特征明显
- 搜索行为和引用样式可能更偏百度系
- 重点保留来源结构和搜索结果元信息

### 通义千问

- 阿里生态偏好更明显
- 结果来源可能偏阿里系内容
- 重点保留平台特有的搜索结果与排序

## 数据模型建议

### 原始采集表

```sql
CREATE TABLE raw_collection_records (
    id VARCHAR(36) PRIMARY KEY,
    trace_id VARCHAR(64) NOT NULL,
    platform VARCHAR(50) NOT NULL,
    provider_model VARCHAR(100),
    query_intent VARCHAR(20),
    user_query TEXT NOT NULL,
    search_triggered BOOLEAN,
    search_query TEXT,
    response_text LONGTEXT,
    citations JSON,
    reasoning_process LONGTEXT,
    search_results JSON,
    token_usage JSON,
    finish_reason VARCHAR(50),
    request_params JSON,
    request_id VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    raw_response JSON,
    INDEX idx_platform (platform),
    INDEX idx_trace_id (trace_id),
    INDEX idx_intent (query_intent),
    INDEX idx_created (created_at)
);
```

### 平台配置表

```sql
CREATE TABLE platform_configs (
    platform VARCHAR(50) PRIMARY KEY,
    config JSON NOT NULL,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 分析结果表

```sql
CREATE TABLE analysis_results (
    id VARCHAR(36) PRIMARY KEY,
    raw_record_id VARCHAR(36) NOT NULL,
    analysis_type VARCHAR(50) NOT NULL,
    result JSON NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (raw_record_id) REFERENCES raw_collection_records(id)
);
```

## 实施流程

```text
输入准备
  ↓
意图识别
  ↓
平台配置选择
  ↓
原生搜索执行
  ↓
原始响应归档
  ↓
响应解析与结构化
  ↓
独立分析输出
```

### 阶段 1：先把原始结果存住

优先把平台响应完整接入并落库，先确保“不丢数据”。

验收标准：

- 能保存完整原始响应
- 能保留平台返回的关键字段
- 解析失败时不丢原文

### 阶段 2：再做平台解析

针对不同平台建立解析规则，提取引用、搜索状态和元数据。

验收标准：

- 至少覆盖主要平台
- 解析失败可降级
- 不影响原始留存

### 阶段 3：对齐平台原生行为

通过手工对照 UI 搜索，调整平台参数和捕获策略，让采集结果更像平台原生输出。

验收标准：

- 结果更接近平台 UI
- 引用和来源结构更一致
- 平台间差异保留真实特征

### 阶段 4：分析层迁移

让既有分析逻辑基于原始采集数据运行，彻底切断“分析反向污染采集”的路径。

验收标准：

- 分析只读原始记录
- 分析结果独立存储
- 原始记录不被重写

## 验证标准

### 定性验证

如果方案生效，应当看到：

- 采集结果更像真实用户在目标平台中看到的搜索结果
- 引用、搜索状态、分析过程和来源网页都能完整保留
- 不同平台的行为差异更真实，而不是被统一逻辑抹平
- 后续分析更可信，因为证据链是完整的

### 定量验证

建议建立人工对照样本集：

- 选 10 到 20 个典型问题
- 分别在平台 UI 和采集系统中执行
- 对比以下维度：
  - 最终回答内容
  - 引用和来源列表
  - 是否触发搜索
  - 搜索结果结构
  - 原始元数据完整度

建议验收目标：

- 结果内容相似度达到可接受阈值
- 引用和来源链路能对上
- 搜索行为差异符合平台原生特征
- 原始字段留存率接近 100%

## 风险与应对

### 1. 平台行为变化

平台会更新搜索策略、引用格式和返回字段，导致结果漂移。

应对：

- 保留平台配置版本
- 定期对照平台 UI 验证
- 发现漂移后快速更新解析规则

### 2. API 与 UI 不一致

如果使用的接口和平台前端行为不同，结果可能不完全一致。

应对：

- 记录采集来源是 API 还是 UI
- 重点平台优先验证 UI 行为
- 在报告中显式标记差异

### 3. 原始数据过大

完整保存会带来更多存储压力。

应对：

- 原始响应和结构化解析分层存储
- 允许历史归档
- 对大字段单独压缩或对象化存储

### 4. 解析规则不稳定

平台格式变化后，解析器可能失效。

应对：

- 解析失败不阻断采集
- 永远保留 raw_response
- 解析规则独立维护和回归测试

## 结论

这份方案的重点已经明确为两件事：

1. **尽可能复现平台自有搜索的原生结果**
2. **把一次搜索的全部内容完整留下来**

搜索提示词采集不再是重点。只要采集层尽量像平台，原始结果留存完整，后续分析就有足够可信的证据基础。
