# 真实用户仿真采集实施任务拆解

本文档把 [技术方案](./user-like-llm-collection-tech-spec.md) 拆成可执行任务，方便按模块实施和验收。

## 1. 实施目标

- 优先保证大模型平台搜索结果的一致性校准能力。
- 完整保留平台返回的原始数据，供后续分析使用。
- 保持前端现有分析结果的内容、模块和交互不变。
- 让采集、解析、分析三层明确分离。

## 2. 任务分组

### A. 数据库与模型

#### A1. 扩展平台响应记录表

目标：
- 给 `platform_response_records` 补齐原始留存字段。

任务：
- 增加 `search_triggered`
- 增加 `search_query`
- 增加 `reasoning_process`
- 增加 `search_results`
- 增加 `search_metadata`
- 增加 `request_params`
- 增加 `request_id`
- 增加 `raw_response`
- 增加 `raw_response_text`

验收：
- 单条平台响应能完整保存原始信息。
- 解析失败时仍能保留原文和 raw payload。

#### A2. 新增平台配置表

目标：
- 独立管理各平台原生搜索行为配置。

任务：
- 新增 `platform_configs`
- 支持配置搜索倾向、参数和引用提取规则
- 支持版本化和回滚

验收：
- 平台配置可读、可改、可回滚。

#### A3. 评估是否新增 raw_collection_records

目标：
- 判断是否需要把“原始归档”和“分析响应记录”进一步拆分。

任务：
- 先评估扩展现有 `platform_response_records` 是否足够。
- 如果字段继续膨胀，再新增 `raw_collection_records`。

建议：
- 短期先扩展现有表。
- 中期再决定是否抽离新表。

### B. 后端采集

#### B1. 平台配置读取

目标：
- 让适配器在执行查询前读取平台配置。

任务：
- 为适配器增加 `get_default_config()` 或等价读取能力
- 查询前加载平台配置
- 将配置注入请求构造逻辑

验收：
- 每个平台能按自己的配置发起请求。

#### B2. 平台适配器统一增强

目标：
- 让适配器返回更完整的 `PlatformResponse`。

任务：
- 保留现有 `query()` 接口
- 补充原始元数据输出
- 保留 citations、token、model、finish reason
- 保留平台原始 payload

验收：
- 单次响应对象能覆盖后续归档和分析所需字段。

#### B3. 平台特殊流式/多轮处理

目标：
- 处理 Kimi 等需要工具调用或多轮 echo 的平台。

任务：
- Kimi 单独处理 `web_search`
- 支持 tool call 过程保存
- 合并最终回答和中间搜索元信息

验收：
- 多轮平台也能落完整原始响应。

#### B4. 审计流程接入

目标：
- 把平台一致性校准纳入现有审计流程。

任务：
- 审计任务发起平台查询
- 保存原始响应
- 继续执行现有分析链路

验收：
- 审计任务完成后，原始响应和分析结果都可追溯。

### C. 响应解析

#### C1. 引用提取器

目标：
- 从不同平台响应中提取 citations。

任务：
- 支持数字引用、来源字段、markdown 链接等格式
- 平台维度配置提取规则

验收：
- 至少覆盖当前主流平台格式。

#### C2. 搜索状态提取

目标：
- 识别是否触发搜索以及搜索词是什么。

任务：
- 从平台返回中识别 search 开关
- 提取 search query / search metadata

验收：
- 能区分“未搜索、搜索但未返回关键词、返回完整关键词”三种情况。

#### C3. 解析降级

目标：
- 解析失败不阻断采集。

任务：
- 解析异常捕获
- 记录 parse_error
- 保留 raw_response 和 response_text

验收：
- 解析失败时系统仍可完成审计流程。

### D. 原始归档

#### D1. 原始响应落库

目标：
- 把平台返回完整保存到归档层。

任务：
- 保存 response_text
- 保存 raw_response
- 保存 raw_response_text
- 保存 citations 和 search metadata

验收：
- 单次响应可完整还原。

#### D2. 去重与引用关系

目标：
- 避免同一次审计中的重复响应污染分析。

任务：
- 保留 `audit_id + prompt_id + platform` 唯一约束
- 让分析表通过外键引用原始记录

验收：
- 同一响应不会被重复写入多条原始记录。

### E. 分析层

#### E1. 保持现有分析能力

目标：
- 不改变前端现有分析结果口径和模块。

任务：
- 品牌提及分析继续基于原始记录
- 来源权威分析继续基于原始记录
- 竞品/趋势/建议流程保持现有输出结构

验收：
- 前端现有展示无需为这次改动做结构性调整。

#### E2. 分析只读原始数据

目标：
- 避免分析结果污染原始采集层。

任务：
- 分析任务只读 `platform_response_records`
- 分析结果落到 `response_analyses` 或现有分析表
- 不回写原始记录

验收：
- 原始记录保持不可变。

### F. 平台一致性校准

#### F1. 平台参数校准

目标：
- 让采集结果更接近目标平台 UI。

任务：
- 为每个平台调参数
- 记录默认搜索倾向
- 校准引用与结果偏好

验收：
- 平台 UI 对照差异可接受。

#### F2. 对照样本集

目标：
- 建立一致性验证样本。

任务：
- 准备 10 到 20 个典型 query
- 覆盖事实、选购、对比、方案、解释类问题
- 保留平台 UI 结果样本

验收：
- 样本可重复比较平台差异。

### G. 前端

#### G1. 前端保持现状

目标：
- 不改变前端现有分析结果模块和内容。

任务：
- 暂不新增平台原始对照页面
- 暂不新增原始响应可视化入口
- 继续沿用现有报告、建议、趋势展示

验收：
- 前端用户仍只看到现有分析结果。

#### G2. 如需微调，仅限文案或标签

目标：
- 避免前端用户误解这次改动的范围。

任务：
- 如有必要，仅更新少量说明文案
- 不改模块结构

验收：
- 用户不会把后台一致性校准误认为新的前端功能。

### H. 测试

#### H1. 单元测试

目标：
- 覆盖配置、解析和归档核心逻辑。

任务：
- 配置读取测试
- citation 提取测试
- search 状态提取测试
- raw_response 存储测试

验收：
- 核心函数有确定性测试。

#### H2. 集成测试

目标：
- 覆盖审计全链路。

任务：
- 审计创建到结束
- 单平台失败不影响其他平台
- 归档和分析双写边界验证

验收：
- 审计任务全链路可跑通。

#### H3. 对照测试

目标：
- 验证平台一致性。

任务：
- 与平台 UI 手工结果对照
- 对比引用链路和搜索状态
- 记录差异原因

验收：
- 能稳定复现平台差异。

## 3. 推荐执行顺序

### 第一阶段

1. 数据库字段扩展
2. 平台配置表
3. 适配器配置读取

### 第二阶段

1. 原始归档落库
2. 引用提取
3. 搜索状态提取
4. 解析降级

### 第三阶段

1. 平台一致性校准
2. 对照样本集
3. 结果差异复盘

### 第四阶段

1. 分析层收敛
2. 保证只读原始数据
3. 保持前端现有模块不变

## 4. 风险点

- 平台返回格式变化会影响解析规则。
- API 和 UI 结果可能不一致。
- 原始字段增加后，表结构可能继续膨胀。
- 前端如果误解范围，可能误以为需要新页面或新模块。

## 5. 验收口径

最终通过标准：

- 平台搜索结果一致性能力可验证。
- 原始响应完整落库。
- 分析层只读原始数据。
- 前端现有分析结果内容和模块保持不变。
- 用户无需感知平台一致性校准细节。
