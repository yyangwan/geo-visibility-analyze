# 真实用户仿真采集方案 1 页版

## 一句话目标

模拟大模型自有平台的原生搜索行为，例如豆包 APP、DeepSeek APP/网页、Kimi、文心一言、通义千问等，让采集结果尽可能接近真实用户在这些平台里实际看到的搜索结果，同时完整保存一次搜索的全部原始内容供后续分析复用。

## 核心原则

- **平台原生优先**：尽量按平台默认搜索行为执行，不人为扭曲结果。
- **原始结果完整留存**：保留回答、搜索状态、分析过程、引用网页、来源元数据、token、模型信息和平台返回的全部原始字段。
- **采集与分析分离**：原始数据只保存，不重写；分析结果独立存储，不回写原始记录。
- **提示词非重点**：搜索提示词继续沿用现状，即 AI 生成或人工录入，不作为本方案重点。

## 方案结构

### 1. 平台配置层

目的：把每个平台的原生搜索特征单独配置出来，避免统一逻辑抹平平台差异。

配置内容：
- 默认是否搜索、搜索强度、搜索触发倾向
- temperature、top_p、max_tokens 等参数
- 引用格式、搜索状态、平台特有字段

作用：
- 让采集行为更像真实用户在对应平台上的搜索
- 平台变更时只需调整配置，不用重写整条链路

### 2. 采集执行层

目的：按平台原生方式发起一次搜索，并拿到尽量接近 UI 结果的响应。

要求：
- 输入保持用户原话
- 默认单轮请求，避免历史上下文污染
- 不附加结构化分析 prompt
- 平台返回什么，就尽量原样接住什么

### 3. 原始归档层

目的：保存“用户看到的第一手结果”，做到完整、可追溯、不可逆改写。

建议保留字段：
- `user_query`
- `search_triggered`
- `search_query`
- `response_text`
- `citations`
- `reasoning_process`
- `search_results`
- `token_usage`
- `finish_reason`
- `request_params`
- `raw_response`

要求：
- 不重写
- 不聚合
- 不二次生成
- 解析失败也不丢原文

### 4. 响应解析层

目的：从原始响应中提取结构化信息，供分析使用。

职责：
- 识别平台响应格式
- 提取引用和来源
- 提取搜索状态和搜索关键词
- 提取平台特有元数据
- 解析失败时降级保留原始响应

### 5. 分析层

目的：基于已保存的原始数据做后续分析。

保留能力：
- 品牌提及分析
- 来源权威分析
- 内容结构分析
- 战略分析
- 竞品对比

边界：
- 只能读原始记录
- 分析结果独立存储
- 不回写原始采集数据

## 数据链路

```text
用户原话 / 手动录入 / AI 生成
          ↓
平台配置选择
          ↓
原生搜索执行
          ↓
完整原始归档
          ↓
响应解析
          ↓
独立分析输出
```

## 落地顺序

1. **先存住原始结果**
   - 先接入并落库完整原始响应，保证不丢数据。
2. **再做平台解析**
   - 为主要平台补齐引用、搜索状态和元数据解析。
3. **再对齐平台原生行为**
   - 用平台 UI 做人工对照，持续调参让结果更像真实搜索。
4. **最后迁移分析层**
   - 让现有分析逻辑只依赖原始采集记录。

## 验收标准

- 采集结果更接近平台 UI 搜索结果
- 引用、搜索状态、分析过程和来源网页都能完整保存
- 不同平台的行为差异能被保留
- 原始字段留存率接近 100%
- 分析层不污染原始采集层

## 风险与应对

- **平台行为变化**：保留平台配置版本，定期对照 UI 验证。
- **API 与 UI 不一致**：记录采集来源，重点平台优先验证 UI 行为。
- **原始数据过大**：原始响应与结构化解析分层存储，支持归档。
- **解析规则失效**：解析失败不阻断采集，永远保留 `raw_response`。

## 结论

这套方案只抓两件事：

1. 尽可能复现平台自有搜索的原生结果
2. 把一次搜索的全部内容完整留下来

搜索提示词采集不再是重点。只要采集层像平台、原始结果留得全，后续分析就有可信证据基础。
